# Website Information Architecture

## Purpose

This document defines the information architecture for the personal portfolio website. It provides a durable content and navigation reference for a professional, engineering-oriented portfolio.

## Website Goals and Target Audience

The website should establish professional credibility, make engineering work easy to evaluate, and provide clear routes to deeper technical evidence and contact.

Primary audiences are hiring managers, engineering leaders, technical recruiters, prospective collaborators, and peers. Each audience should be able to quickly understand the portfolio owner's capabilities, review selected work at an appropriate depth, and find a practical next action.

## Main Navigation

```text
Home
About
Projects
Technical Notes
Resume
Contact
```

The primary navigation is consistent across the site. Project detail pages and technical-note articles are reached from their respective index pages and include a clear return path to the parent section.

## Page Responsibilities

| Page | Purpose | Content responsibility |
| --- | --- | --- |
| **Home** | Provide a fast professional introduction and guide visitors to the most important evidence. | Value proposition, selected projects, core expertise, current focus, and primary calls to action. |
| **About** | Explain professional identity and working approach. | Concise background, engineering interests, principles, capabilities, and a human but professional narrative. |
| **Projects** | Serve as the index of portfolio case studies. | Filterable or scannable project summaries, technology context, status, and links to detailed case studies. |
| **Technical Notes** | Demonstrate ongoing technical thinking beyond individual projects. | Focused articles, design notes, experiments, lessons learned, and references to relevant projects. |
| **Resume** | Provide a structured summary for professional evaluation. | Experience, education, skills, selected achievements, and a downloadable or printable resume where appropriate. |
| **Contact** | Convert interest into a clear, low-friction conversation. | Professional contact methods, collaboration interests, availability guidance, and external professional links. |

## Project Case Study Template

Every project case study should use a consistent narrative so visitors can assess both the work and the engineering judgment behind it.

1. **Overview** - Project summary, role, scope, status, and relevant technologies.
2. **Problem** - User, business, or technical need the project addresses.
3. **Engineering Challenge** - Constraints, risks, unknowns, trade-offs, and the hard technical questions.
4. **Solution Architecture** - System boundaries, key components, data flow, major decisions, and diagrams where useful.
5. **Implementation** - Important implementation choices, tools, workflows, and representative technical details.
6. **Validation** - Testing, measurement, review, or other evidence that the solution meets its intent.
7. **Outcome** - Results, delivered value, limitations, and lessons learned.
8. **Future Development** - Planned improvements, open questions, and possible extensions.

## Homepage Content Hierarchy

The home page prioritizes fast orientation before deeper exploration.

1. **Professional introduction** - Name, engineering focus, short value proposition, and primary action.
2. **Selected work** - A small set of strongest project case studies, led by outcome and technical relevance.
3. **Core capabilities** - Concise technical domains, methods, and tools supported by project evidence.
4. **Current focus** - Active areas of learning, research, or project development.
5. **Technical notes** - Recent or featured writing that demonstrates technical depth and continuity.
6. **Professional summary** - Link to the resume and brief credibility signals.
7. **Contact call to action** - A direct route to start a professional conversation.

## Navigation Design Principles

- **Clarity before novelty:** Use familiar labels that state the destination directly.
- **Stable global structure:** Keep the main navigation visible and consistent across routes.
- **Progressive disclosure:** Present concise summaries first, then allow visitors to move to project details, source material, or technical notes.
- **Evidence-led paths:** Prioritize routes from claims of capability to projects, validation, and technical detail.
- **Low cognitive load:** Limit top-level navigation to the six primary sections and avoid duplicate destinations.
- **Accessible and responsive behavior:** Support keyboard navigation, clear focus states, readable labels, and usable small-screen navigation.
- **Context preservation:** Use page titles, active navigation states, and breadcrumbs or parent links where nested content benefits from orientation.
